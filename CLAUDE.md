# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

"Fretze" (`index.html`, title "Fretze – Trainingsbuch"; named "Eisernes Log" before the 2026-07-08 rebrand to "Fretze pumpt", shortened to "Fretze" on 2026-07-09 — see `00_Projektsteuerung/PROJEKTLOG.md` for history) is a single-user Push/Pull/Legs workout tracker, built as an installable PWA. It is entirely client-side, in German. **Since v1.23.0, the visible in-app header (`<h1>`) shows "Fretze pumpt" again** (explicit user preference after live-testing the shortened name) — deliberately only the header, not a full revert: the `<title>` tag, `howto.html`'s prose, and the app's formal name everywhere else stay "Fretze". Not a bug if the two diverge.

## Versioning (process rule, active since v1.1.0)

Every change gets a version bump and a `00_Projektsteuerung/PROJEKTLOG.md` entry — no exceptions, this was an explicit user request.

- Bump `APP_VERSION` in `index.html` (semantic-ish: PATCH for small fixes, MINOR for features, MAJOR for breaking/rewrite-level changes).
- Bump `CACHE_NAME` in `sw.js` to match (e.g. `fretze-pumpt-v1.2.0`) — this is not just bookkeeping, it's what makes the PWA actually update on installed devices (see the auto-update note below). The two must move together.
- Add a dated, versioned entry to `00_Projektsteuerung/PROJEKTLOG.md` describing what changed, before/alongside deploying.

## User-facing documentation (`howto.html`, process rule, active since v1.5.1)

`00_Projektsteuerung/PROJEKTLOG.md` is a developer-facing changelog (for future Claude sessions, not end users). `howto.html` is the separate, plain-German **user-facing** how-to page, linked from the app header ("❓ Anleitung") — explicit user request: **every change relevant to what the user sees or does must also update `howto.html`**, not just `PROJEKTLOG.md`. Examples of "relevant": a new/changed feature, a UI element that moves or behaves differently, a new settings/toggle. Not relevant: internal refactors, data-model changes with no visible effect, doc/versioning-only commits. `howto.html` reads the same `localStorage` theme key as `index.html` to match dark/sepia automatically, and is listed in `sw.js`'s `APP_SHELL` for offline use — keep both in sync if it's ever renamed.

## Commands

There is no build, lint, or test tooling — no `package.json`, no dependencies. The app is plain HTML/CSS/JS.

- **Develop**: edit `index.html` directly and reload in a browser.
- **Test PWA/offline behavior**: the service worker (`sw.js`) only registers on a real HTTP origin, not `file://`. Serve the folder locally, e.g. `npx serve` or `python -m http.server`, then open it.
- **Deploy**: pushing to `main` on GitHub is the deploy — GitHub Pages serves this repo directly (no `gh-pages` branch, no build step). Live at the repo's Pages URL. Only push to `main` when explicitly asked to deploy.
- **Deploy the Gemini proxy worker**: `cd 03_Tools/worker && wrangler deploy` (requires `wrangler secret put GEMINI_API_KEY` once beforehand). This is separate from the GitHub Pages deploy — the worker lives on Cloudflare, not in this repo's Pages site.

## Architecture

Everything lives in one file, `index.html`: inline `<style>` block plus a single inline `<script>` at the bottom (~680 lines total). No framework, no bundler. Supporting files are `manifest.json` (PWA metadata), `sw.js` (cache-first service worker caching the app shell), `icon-*.png`, `logo-mark-dark.png`/`logo-mark-sepia.png` (header logo, transparent PNGs; see Ordnerstruktur above for the source file), and `howto.html` (standalone user-facing how-to page, see the User-facing documentation section below). All must stay in the app shell list in `sw.js` (`APP_SHELL`) — a new asset referenced from `index.html` but missing there won't be available offline.

**Header logo is theme-aware, unlike the app icons**: `render()` picks `logo-mark-dark.png` or `logo-mark-sepia.png` based on `state.theme`, mirroring the existing `theme-color`-meta swap. Two separate PNGs exist (rather than one raster recolored to the shared `--rust`) because a single color didn't read well in both contexts: dark theme needed a brighter/more saturated orange (`#F97316`) for the fine linework to stay legible against the near-black `--bg`, while sepia keeps the theme's own `--rust` (`#A8461E`), which already contrasts fine on the light background. The app-icon PNGs (`icon-192.png`/`icon-512*.png`) are NOT theme-aware — OS home screens don't know about in-app theme state, so there's necessarily one fixed appearance regardless of `state.theme`. Since v1.14.0 that fixed appearance is the **sepia** styling (cream `#F2E8D5` background, `logo-mark-sepia.png`'s rust linework, generated at a much higher fill fraction — ~86% of the canvas for `icon-192.png`/`icon-512.png`, ~74% for the `icon-512-maskable.png` safe zone — than the original export, which left a lot of empty padding and read as "too small" per user feedback) — chosen because sepia is also the app's actual default theme (`loadState()`'s fallback). `manifest.json`'s `background_color`/`theme_color` and `index.html`'s static initial `<meta name="theme-color">` were updated to the same sepia palette (`#F2E8D5`/`#A8461E`) to match, avoiding a dark splash-screen flash against the now-cream icon. Regenerating these icons requires re-deriving the logo's tight bounding box from `logo-mark-sepia.png` (or `-dark.png`) and re-detecting the existing rounded-corner cutout shape from the *previous* icon file as a mask — see the (uncommitted, session-scratch) Python regeneration script for the exact approach if this needs doing again.

**PWA update mechanism**: `sw.js` uses a cache-first strategy, which means an installed/home-screen copy will keep serving whatever it last cached until the service worker itself gets updated — browsers only re-check `sw.js` for byte-level changes, they don't know `index.html` changed on their own. That's why `CACHE_NAME` must change every release (see Versioning below): a different string makes `sw.js`'s bytes differ, which is what triggers the browser to install the new worker (`skipWaiting()` + `clients.claim()` in the code make it take over immediately), delete the old-named cache, and re-fetch the app shell fresh. Skipping the `CACHE_NAME` bump on a release means installed devices silently keep the old version.

**Naming stability**: `STORAGE_KEY` (`eisernes-log-v1`) and the Cloudflare Worker name (`eisernes-log-proxy`) were deliberately *not* renamed during the 2026-07-08 rebrand to "Fretze pumpt", nor during the 2026-07-09 shortening to "Fretze" — changing `STORAGE_KEY` would wipe every user's existing `localStorage` data, and renaming the deployed worker has no user-facing benefit. Don't rename these without explicitly flagging the data-loss risk to the user first.

**Data model** (top of the script):
- **Multiple plans (since v1.5.0, 5th added in v1.21.0, 6th in v1.24.0)**: `PLANS` is an array of `{ id, label, days, exercises }` — currently `ppl` (`PPL_DAYS`/`PPL_EXERCISES`, the original 6-day split), `upperLower` (`UL_DAYS`/`UL_EXERCISES`), `fullbody` (`FB_DAYS`/`FB_EXERCISES`), `broSplit` (`BRO_DAYS`/`BRO_EXERCISES`), `calisthenics` (`CALI_DAYS`/`CALI_EXERCISES`, a 4-day bodyweight-only Upper/Lower split, sourced from `01_Recherchen/01_Trainingsplaene/Zuhause-Calisthenics-Trainingsplan.md`), `baender` (`BAND_DAYS`/`BAND_EXERCISES`, a 4-day Upper/Lower split for the "SmartWorkout Elite" resistance-band set — see the dedicated paragraph below) — all sourced from `01_Recherchen/01_Trainingsplaene/`. `DAYS`/`EXERCISES` are `let` (not `const`): they're mutable pointers to the *active* plan's data, reassigned by `applyPlan(planId)`. Every other function in the file (`render()`, event handlers, `getRecommendedDay()`, …) reads `DAYS`/`EXERCISES` normally and doesn't need to know plans exist. **`PLANS` itself stays fixed to the 6 built-ins** (since v1.11.0) — user-created plans live in `state.customPlans` instead, and `getAllPlans()` (`[...PLANS, ...state.customPlans]`) is what `applyPlan()`, the plan dropdown, and the home screen actually iterate; don't read `PLANS` directly anywhere that should also see custom plans.
- **Day/exercise IDs are globally unique across all plans** (e.g. PPL uses `pushA`/`pa1`, Upper/Lower uses `ulUpperA`/`uua1`, never overlapping) — this is *the* mechanism that makes "switch plans without losing data" work: `state.sessions`/`history`/`swaps`/`notes`/`lastWeights` stay flat objects keyed by these IDs, so switching `state.currentPlan` never overwrites another plan's entries; nothing needed migrating. Don't reuse an ID across plans.
- `DAYS` entries have an optional `type` (`"push"`/`"pull"`/`"legs"`) used to look up `WARMUP`/`STRETCHING` content per day-type. Only assigned where the research content genuinely applies (e.g. Upper/Lower's "Lower" days get `"legs"`, Bro-Split's "Schultern"/"Arme" days get none) — `warmupSectionHTML()`/`stretchSectionHTML()` return `""` when there's no matching entry, so a missing `type` just means no section shows, not a crash.
- `EXERCISES`: keyed by day id, each an array of exercises with `id`, `name`, `sets`, `reps` (target range), optional `alts` (swappable alternative exercise names), `core` (boolean flag for core exercises), `compound` (boolean, since v1.12.0 — flags the canonical "big lifts": squat, deadlift/RDL, flat bench, overhead press, pull-up, barbell row, regardless of exact equipment; drives the higher progression percentage in `computeSuggestedWeight`), and `libraryId` (optional, since v1.16.0 — see below). This is the content to edit when changing a training program itself. Only the `ppl` plan's exercises have populated `alts`; the 4 newer built-in plans (`upperLower`/`fullbody`/`broSplit`/`baender`) default to `alts: []` (no researched alternatives yet) — `calisthenics` is the one exception, see its own paragraph below. Custom (user-built) plan exercises get no `alts`/`core`/`compound` — always the defaults/standard progression rate — but *can* have a `libraryId` if built from a database-picked exercise name.
- **Übungsdatenbank / `exerciseLibrary` (since v1.16.0)**: exercise *knowledge* (muscles/steps/mistakes/safety, optionally an `image`) lives in `exercise-library.json` (repo root, sibling to `manifest.json`) — deliberately **not** inlined in `index.html`, so the database can grow without bloating the app bundle or needing an `APP_VERSION` bump per new exercise. Keyed by a stable **content ID** (a slug derived from the exercise name, e.g. `flachbankdruecken-kurzhantel`) that's independent of any plan — this is the key architectural difference from the old `EXERCISE_INFO` (which was keyed by plan-slot `ex.id`, so the same real exercise appearing in multiple plans would've needed its explanation duplicated per plan). Exercise objects link to it via the optional `libraryId` field; `exerciseInfoPanelHTML(ex)` (the single function building the ℹ-panel markup, called from card view/focus mode/library view) resolves it as `exerciseLibrary[ex.libraryId]` — no `libraryId`, or a `libraryId` the library doesn't (yet) have, both degrade gracefully to no ℹ button, same as the old missing-`EXERCISE_INFO`-entry behavior. Loaded once at boot via `loadExerciseLibrary()` (`async`, called *after* the first synchronous `render()` so the app doesn't wait on the fetch to become usable — `exerciseLibrary` starts as `{}` and a failed/offline fetch just leaves it that way, caught and swallowed, no crash). `sw.js` needed **no changes** for this: its existing `fetch` handler already caches any same-origin GET response after first fetch, not just `APP_SHELL` entries, so `exercise-library.json` (and future per-exercise images) get cached opportunistically without bloating the explicit offline app-shell list. All 4 built-in plans now have `libraryId`s populated (`ppl` migrated 1:1 from the old `EXERCISE_INFO`, with one real duplicate — "Beinstrecker" existed twice, verbatim, under two plan-slot IDs — collapsed into a single library entry both slots now point to; Upper/Lower/Ganzkörper/Bro-Split added in v1.17.0 from a second Deep Research batch of 37 exercises, `01_Recherchen/02_Uebungserklaerungen/Uebungsdatenbank-UL-FB-BroSplit.md`, parsed and merged the same way). The library-first design pays off here: the same real exercise repeated across multiple plans (e.g. "Kniebeugen mit der Langhantel" in Upper/Lower, Ganzkörper, and Bro-Split) shares **one** library entry rather than being researched/stored three times. Custom plan exercises still get no `libraryId` unless built from a database-picked name (see the plan-builder note above) — same graceful gap as before. The plan-builder's exercise-name `<datalist>` (`planBuilderExerciseNamesDatalist()`) is sourced from `exerciseLibrary` (not raw plan-exercise names) specifically so picking a suggested name always yields a working ℹ explanation — `libraryIdForName(name)` does the reverse lookup and gets auto-attached to a draft exercise when its name exactly matches a library entry; free-text names simply get no `libraryId`, exactly like today's gap.
- `WARMUP`/`STRETCHING`/`STRETCHING_NOTE`: keyed by day `type`. Sourced from `01_Recherchen/03_Aufwaermen/` and `01_Recherchen/04_Stretching/`; rendered as collapsible sections above/below the exercise cards. None of this — nor the exercise-info panel — is part of `state`/`localStorage`; open/closed state (`infoOpenKey`/`warmupOpen`/`stretchOpen`) is ephemeral like `swapOpenKey`.
- **`calisthenics` plan's `alts` repurposed for progression, not lateral swaps (since v1.21.0)**: every other plan's `alts` lists genuinely different alternative exercises (same muscle group, different movement) for the ⟲ swap feature. The calisthenics research instead frames progression as **difficulty variants of the same exercise** (e.g. knee push-ups → push-ups → archer push-ups), since there's no external weight to add for pure bodyweight training. Rather than building a dedicated progression UI, `CALI_EXERCISES` populates `alts: [easierVariant, harderVariant]` and reuses the existing swap panel as-is — a deliberate, pragmatic repurposing of an existing mechanism rather than new architecture. Only 2 of the plan's exercises (`Klimmzüge im pronierten/supinierten Griff` — pull-ups/chin-ups) have a `libraryId`, reusing the exact same entries already shared by Upper/Lower/Ganzkörper/Bro-Split; every other exercise (the push-up family, Pike Push-ups, Pistol Squats, Nordic Hamstring Curls, Sliding Leg Curls, Bulgarian Split Squats, Inverted Rows, RKC Plank) has none yet — the research document's own biomechanical narrative doesn't include step-by-step Ausführung/Fehler/Sicherheit content in the format `exercise-library.json` needs, so these are a graceful gap (no ℹ button) awaiting a smaller, dedicated follow-up research batch, exactly like UL/FB/BroSplit's exercises before v1.17.0.
- **`baender` plan (since v1.24.0) uses normal weight progression, NOT the calisthenics `alts`-repurposing**: a 4-day Upper/Lower split for the "SmartWorkout Elite" resistance-band set (7 bands 5–25kg, stackable to ~120–160kg combined, plus a bar/handles/door-anchor/ankle-straps), sourced from `01_Recherchen/01_Trainingsplaene/SmartWorkout-Baender-Trainingsplan.md`. Unlike calisthenics, bands allow real continuous resistance progression (combine more/stronger bands) much like free weights, so `BAND_EXERCISES` uses the same `compound`/`weightSuggestions` mechanism as the gym plans instead of repurposing `alts` — `compound: true` is set only on the bar-based canonical lifts (bench/row/overhead-press/squat/RDL/deadlift equivalents), everything else (door-anchor and unilateral variants, isolation moves) is unflagged, matching the existing gym-plan convention. `alts: []` on all 22 exercises (no researched alternatives, same gap as the other non-`ppl` plans). The weight field is meant to hold the currently combined band resistance in kg — there's no dedicated band-combination helper in the UI yet (flagged as a Gemini-feedback wishlist item in `00_Projektsteuerung/MEMORY.md`, not built). **Unlike calisthenics, all 22 exercises have a `libraryId`** (since v1.25.0, `01_Recherchen/02_Uebungserklaerungen/SmartWorkout Biomechanischer Übungskatalog.md`) — a dedicated research pass was needed because band exercises have a materially different (accommodating/progressive) resistance curve than their free-weight namesakes, so the existing free-weight library entries couldn't just be reused; only 2 of `calisthenics`'s exercises share this full-coverage treatment (the pull-up/chin-up entries, reused verbatim from Upper/Lower).

**State & persistence**: a single in-memory `state` object (`currentDay`, `currentPlan`, `sessions`, `history`, `swaps`, `notes`, `lastWeights`, `lastCompletedDay`, `lastCompletedDate`, `theme`, `textSize`, `gyms`, `currentGymId`, `gymLastWeights`, `gymHistory`, `journal`, `customPlans`, `planDraft`, `weightSuggestions`, `gymWeightSuggestions`) is loaded from and persisted to `localStorage` under key `eisernes-log-v1` (debounced ~300ms via `persist()`). `textSize` (`"normal"`/`"gross"`/`"xgross"`) drives the `--text-zoom` CSS variable (via `data-textsize` on `<html>`, same pattern as `data-theme`) applied as `zoom` on `#app`/`#rest-timer-bar` — scales the whole UI, not just font-size declarations, so no per-rule CSS changes were needed. `zoom` needs Firefox 126+ (mid-2024); older Firefox just silently ignores it (graceful, no crash). `applyPlan(state.currentPlan)` must run after `state` is loaded/replaced (on initial load and after Import) so `DAYS`/`EXERCISES` point at the right plan before anything renders — both call sites also snap `state.currentDay` back to `DAYS[0].id` if it doesn't belong to the now-active plan. There is no backend — the Export/Import buttons in the header serialize/restore this state as a JSON file, and that's the only backup mechanism.

**Rendering**: no virtual DOM — `render()` rebuilds `#app`'s `innerHTML` from `state` on every state change (day switch, set marked done, warmup-set toggle, swap, finish session). One exception: raw text inputs (set weight/reps, notes) update `state` directly in an `input` listener without calling `render()`, to preserve focus/cursor position while typing; only `persist()` runs there.

**Events**: everything is delegated through three top-level listeners (`click`, `input`, `change`) on `document`, dispatching on `data-role` attributes (e.g. `data-role="set-done"`, `data-role="swap-toggle"`, `data-role="finish"`) rather than per-element handlers. New interactive elements should follow this pattern: give them a `data-role` (plus `data-day`/`data-exid`/`data-idx` as needed) and add a branch in the relevant delegated listener.

**Session flow**: `finishSession(dayId)` records the last completed set per exercise into `state.history` (shown as "Letztes Mal: ..."), snapshots current weights into `state.lastWeights` (used by `buildInitialSets` to prefill next time), resets the day's sets, and stamps `lastCompletedDay`/`lastCompletedDate` (used by `renderTodayBanner()`/`getRecommendedDay()` to suggest the next day in the PPL rotation on the home banner).

**Exercise swapping**: `state.swaps[`${day}:${exId}`]` maps an exercise slot to a chosen alternative name (from that exercise's `alts` list, or an AI-suggested name); the original exercise's `sets`/`reps` targets and history are still keyed by the original `ex.id`, only the displayed name changes.

**Gym-Standorte (since v1.7.0)**: `state.gyms` (`[{id, name}]`) + `state.currentGymId` let the user manually pick which studio they're training at (no GPS — deliberate, see PROJEKTLOG v1.7.0) when weights/equipment differ between locations (e.g. Wellhub/Urban Sports Club users who train at MCFit one day and Fitness First the next). `currentGymId === null` ("Kein Gym") reproduces today's exact behavior via the original flat `state.lastWeights`/`state.history`; a non-null `currentGymId` routes reads/writes through `state.gymLastWeights[gymId]`/`state.gymHistory[gymId]` instead — both weight-prefill *and* the "Letztes Mal" line are gym-scoped, with no fallback to the flat data on a gym's first use (deliberately isolated, to avoid exactly the cross-studio weight confusion this feature exists to prevent). All reads/writes go through four accessor functions (`getLastWeights`/`setLastWeights`/`getHistoryEntry`/`setHistoryEntry`) rather than touching `state.lastWeights`/`state.history`/`state.gymLastWeights`/`state.gymHistory` directly — `buildInitialSets` and `finishSession`/`render` call these, not the raw state fields. Deleting a gym only removes it from `state.gyms`; its `gymLastWeights`/`gymHistory` entries are left orphaned rather than deleted (no destructive data loss in a single-user app).

**Set-level flags (since v1.8.0/v1.9.0)**: each set row (`{weight, reps, isWarmup, couldDoMore, done}`) carries two independent boolean toggles rendered side by side in `setsRowsHTML()` — `isWarmup` (🔥, since v1.8.0, replaced the old 0/1/2/3+ RIR chips; warmup-marked sets are excluded from `finishSession`'s "last set" history pick and from the footer's Arbeitssätze count) and `couldDoMore` (↑, since v1.9.0, a deliberately simplified "wären mehr Wdh. möglich gewesen?" yes/no replacing the old RIR granularity). `couldDoMore` is capture-only for now — nothing reads it yet; it's meant to feed a future weight-progression suggestion (Phase D of the roadmap in `00_Projektsteuerung/PROJEKTLOG.md`/the session's plan file), deliberately scoped out of this pass.

**Auto-mark-done on reps entry (since v1.23.0)**: entering a value in a set's `reps` field and leaving/confirming it (the delegated `change` listener, not `input` — so it fires once the value is committed, not on every keystroke) automatically marks that set `done` (✓, green) and starts the rest timer, saving the extra tap on the checkbox. Only fires the first time (`if (!s.done)`) so correcting the reps afterward doesn't re-trigger the timer; the manual done-checkbox still works unchanged for edge cases (e.g. marking a set done with no rep count, or undoing an auto-mark).

**Trainingsmodus / Fokus-Ansicht (since v1.9.0)**: an alternate single-exercise full-screen rendering path, entered via the "▶ Trainingsmodus starten" button (`data-role="focus-enter"`) in the card view. Ephemeral UI state, never persisted (same convention as `swapOpenKey`/`restTimer`): `focusModeActive`, `focusQueue` (exercise IDs for the current day, snapshotted at entry, reordered by skip), `focusPos`, `focusInfoOpen`. `render()` sets the theme/textsize attributes unconditionally, then early-exits to `renderFocusMode()` when `focusModeActive` is true, before reaching the normal card-list body — so both paths share the same top-of-`render()` setup but diverge on the actual content. `setsRowsHTML(day, ex, sets)` and `exerciseInfoPanelHTML(ex)` were extracted out of the card-view render loop specifically so both views render exercise sets/info identically and can't drift apart; card-view and focus-mode both call them. Skipping (`skipCurrent()`) moves the current exercise from its position to two slots later in `focusQueue` (`splice` out, `splice` back in at `min(pos+2, length)`) — with fewer than 2 remaining exercises it just moves to the end. The last exercise's "Weiter" button is replaced by the existing `data-role="finish"` button (reused, not duplicated); its click handler sets `focusModeActive = false` *before* calling `finishSession()`, so `finishSession`'s own internal `render()` call lands back in card view rather than re-rendering a stale/out-of-range focus position. Exercise swapping and the AI-suggest panel are deliberately unavailable in focus mode v1 (card-view only) — a user needs to exit focus mode to tap ⟲. A reload mid-focus-mode loses only the ephemeral position, never logged set data (that always lives in `state.sessions`).

**Tagebuch (since v1.10.0)**: `state.journal` is a flat, append-only array of session entries (`{date, timestamp, planId, dayId, dayLabel, gymId, exercises: [{exId, name, sets: [{weight, reps, isWarmup, couldDoMore}]}]}`), pushed by `addJournalEntry()` from `finishSession()` — only sets with an actual weight/reps entry are kept, and a session with zero entered data across all exercises produces no entry at all. `name` is the *displayed* (post-swap) name at the time, not necessarily `ex.name`. Deliberately **not** gym-scoped like `lastWeights`/`history` — the journal is a single chronological log; `gymId` is stored per entry purely for display, since a personal record is a personal record regardless of which studio it happened in. Personal records and progress-chart data are **derived, not stored**: `computePersonalRecords()` scans `state.journal` for the heaviest non-warmup weight ever logged per `exId` (PR definition is deliberately the simplest possible — heaviest weight regardless of rep count — precisely because it's computed on demand rather than persisted, so the definition can change later without any migration); `journalPointsFor(exId)`/`sparklineSVG(points)` build a small dependency-free inline-SVG polyline for the same underlying data (no charting library in the project). The journal view (`renderJournalView()`, entered via the header's "📔 Tagebuch" button, `data-role="journal-enter"`) follows the same ephemeral-state/early-return-in-`render()` pattern as Trainingsmodus (`journalViewActive`, `journalTab` — `"verlauf"`, `"rekorde"`, or `"volumen"` since v1.18.0, see below), and is read-only in this version: no editing or deleting individual journal entries.

**Wochenvolumen-Dashboard (since v1.18.0)**: a third Tagebuch tab (`journalTab === "volumen"`, `journalVolumeHTML()`) showing trained sets per muscle group for the current ISO calendar week (Monday–Sunday), against the 12–20 sets/muscle/week rule of thumb from `01_Recherchen/01_Trainingsplaene/Trainingswissenschaft-Volumen-Frequenz-Splits.md`. Since `state.journal` entries only store the plan-slot `exId` (not `libraryId`), `findExerciseDefById(exId)` re-finds the original exercise definition across all plans (built-in + custom) by scanning for a matching `id` — safe because IDs are globally unique across plans (see above); returns `null` if the exercise no longer exists (e.g. removed from an edited custom plan), which `computeWeeklyVolume()` then tallies as "nicht zuordenbar" rather than silently dropping. `computeWeeklyVolume()` follows the same derive-don't-persist convention as `computePersonalRecords()`: filters `state.journal` to entries with `timestamp >= getWeekStart(new Date())`, counts 1 point per non-warmup set toward its resolved muscle group. Deliberately **no** multi-joint/single-joint fractional volume counting toward secondary/synergist muscles in this version (that's flagged as an open debate in `01_Recherchen/01_Trainingsplaene/Trainingswissenschaft-Uebungsauswahl.md`, not adopted, to keep v1 simple) — every non-warmup set counts as exactly 1.0 toward its primary muscle group only.

**Startbildschirm / eigene Pläne (since v1.11.0)**: `homeScreenActive` is the one ephemeral view flag that starts **`true`** (not `false` like every other view flag) — it's what makes the home screen appear automatically on the very first `render()` call after page load, without any user action; a header "🏠 Start" button (`data-role="home-enter"`) re-opens it later. `renderHomeScreen()` offers "Weiter mit [Plan] – [Tag]" (reuses `getRecommendedDay()`), the plan list from `getAllPlans()` (🗑 delete button shown only for entries whose id starts with `custom_`), and an entry into the plan builder. The plan builder (`renderPlanBuilder()`, `planBuilderActive`) edits `state.planDraft` — this one is **deliberately persisted, not ephemeral** (a half-built multi-day, multi-exercise plan is real user input that a stray reload shouldn't destroy), created via `newPlanDraft()` and mutated by `draftAddDay`/`draftRemoveDay`/`draftAddExercise`/`draftRemoveExercise`. The exercise-name field uses a plain `<datalist>` (`planBuilderExerciseNamesDatalist()`, populated from `exerciseLibrary` since v1.16.0 — see below) as the "picker" — free text still works, the datalist is purely a suggestion aid. `savePlanDraft()` validates via `validatePlanDraft()` (name + ≥1 day + every day has ≥1 exercise with name/sets/reps) and refuses to save an incomplete draft (shows a toast, keeps the draft intact) rather than silently discarding it; on success it calls `buildPlanFromDraft()`, which mints collision-proof IDs for anything lacking one (`custom_<Date.now()>` for a brand-new plan, `${planId}_day<n>`/`${planId}_ex<n>` for days/exercises — necessary because `state.history`/`state.lastWeights`/`state.swaps`/`state.notes` are all keyed globally by bare `ex.id` with no existing collision guard), then either pushes into `state.customPlans` (new plan) or replaces the matching entry in place (editing), and switches to that plan. Custom exercises still get no `alts`/`core`/warmup-type content, but *can* get a working ℹ explanation if built from a database-picked name (see `libraryId` below) — no longer a blanket gap.

**Editing an existing custom plan (since v1.15.0)**: `startEditingPlan(planId)` (entry: ✏ button on the home screen, next to 🗑) loads a plan from `state.customPlans` into `state.planDraft`, setting a new `draft.editingPlanId` field and — critically — **preserving each day's and exercise's existing `id`** in the draft (`draftAddDay`/`draftAddExercise` still produce id-less entries for genuinely new rows, unchanged from creation). `buildPlanFromDraft()` now branches on this: entries that already carry an `id` keep it as-is (so `state.history`/`lastWeights`/`swaps`/`notes`/`journal` entries tied to that exercise stay linked); only id-less entries get freshly minted ones, numbered via `nextIndexFor(prefix, existingIds)` starting *above* the highest existing suffix in that plan (not restarting at 1) to avoid colliding with kept IDs. Removing a day/exercise during an edit just omits it from the rebuilt plan on save — its historical data is left orphaned, not cleaned up, matching the same soft-delete convention already used for gym/plan deletion. `renderPlanBuilder()`/`savePlanDraft()` read `draft.editingPlanId` purely for copy ("Plan bearbeiten"/"Änderungen speichern" vs. "Neuen Plan erstellen"/"Plan speichern") and to decide push-vs-replace into `state.customPlans` — the validation and ID-minting logic is otherwise identical between create and edit.

**Übungs-Bibliothek (since v1.15.0)**: `renderLibraryView()` (ephemeral `libraryActive`/`libraryInfoOpenKey`, entry via a "📚 Übungen durchsuchen" button on the home screen — deliberately not another header icon, given the header was just decluttered in v1.13.0) lists every exercise across `getAllPlans()`, grouped by plan then day, reusing `exerciseInfoPanelHTML(ex)` unchanged for the ℹ toggle (no duplicate info-rendering logic). Deliberately has no live-search: an `input` handler that calls `render()` on every keystroke would drop focus mid-type (the exact problem `set-field`/`note` already avoid by skipping `render()`) — the plan/day grouping serves as browsing structure instead for v1.

**Gewichtsprogression (since v1.12.0)**: `finishSession()` checks each exercise's last *work* set (same "last set counts" convention as the history write) — if it was marked `couldDoMore` and its reps meet/exceed the parsed top of `ex.reps` (`parseRepRange(str)`, which only recognizes clean numeric ranges/single numbers like `"6-8"`/`"10"` and deliberately returns `null` — no suggestion — for anything else, e.g. `"12/Seite"`, `"max"`, time/distance formats like `"30-45s"`/`"30-40m"`), it computes and stores a suggestion via `setWeightSuggestion`/`getWeightSuggestion`/`clearWeightSuggestion` (gym-scoped exactly like `lastWeights`/`history`, backed by `state.weightSuggestions`/`state.gymWeightSuggestions`); otherwise it clears any stale suggestion for that exercise. `computeSuggestedWeight(weight, isCompound)` is percentage-based, not a fixed kg step (deliberately chosen over an earlier fixed/gym-configurable-increment design, since a percentage scales automatically with the weight without needing per-exercise equipment classification) — `PROGRESSION_PCT_COMPOUND` (5%) for exercises flagged `compound: true`, `PROGRESSION_PCT_DEFAULT` (2.5%) otherwise, rounded to the nearest 0.5 kg with a guaranteed minimum +0.5 kg bump (so rounding never makes a suggestion a no-op at light weights). The suggestion is **non-destructive**: `historyLineHTML(exId, hist)` (now shared between card view and focus mode, replacing what used to be two separately-duplicated inline history-line blocks) renders a "💡 Vorschlag: …kg — Übernehmen" chip next to "Letztes Mal" when a suggestion exists; the `suggestion-accept` click handler writes the suggested weight into *every* set of that exercise in the current session and clears the suggestion — nothing is ever auto-applied. **Since v1.29.0**, when `state.currentPlan === "baender"`, the chip additionally shows a band-combination hint in parentheses (e.g. "35kg (20+15kg)") via `findBandCombination(targetKg)` — a brute-force search (only 2^7=128 subsets of the SmartWorkout set's 7 bands, `BAND_WEIGHTS = [25,25,20,20,15,10,5]`, trivially fast) for whichever subset sums closest to the target, tie-broken toward fewer bands. Purely a display hint, no interaction — doesn't affect `suggestion-accept`'s behavior.

**Cloud-Sync-Statusanzeige (since v1.29.0)**: `cloudSyncStatus` (`"idle"|"syncing"|"synced"|"error"`) and `lastCloudSyncAt` are ephemeral module-level state (never in `state`/localStorage, same convention as `restTimer`/`modal`), driven by `persist()`'s Firestore push and `reconcileCloudState()` (both call `setCloudSyncStatus(...)` at the relevant points). Shown as a "Cloud-Sync" row in `accountSectionHTML()`'s logged-in branch. Deliberately does **not** trigger a full `render()` on every status change — `persist()` is called from the raw weight/reps `input` listener, which itself deliberately skips `render()` to avoid losing input focus while typing (see "Rendering" above); a `render()` call buried inside `persist()`'s Firestore branch would silently reintroduce that exact bug. Instead `renderCloudSyncStatus()` only updates the single `#cloud-sync-status` DOM node directly (same targeted-update pattern as `renderRestTimer()`/`showToast()`), a no-op if the Profil view isn't currently open (`getElementById` returns `null`) — the next time `renderProfileView()` runs for any other reason, `cloudSyncStatusHTML()` bakes in whatever the latest status is anyway.

**Profil view (since v1.13.0)**: the header used to hold theme/textsize/export/import as individual `icon-btn`s alongside "🏠 Start"/"📔 Tagebuch"/"❓ Anleitung" — on narrow phones that's too many buttons for one row (user-reported via screenshot). `renderProfileView()` (ephemeral `profileActive`, same early-return-in-`render()` pattern as every other view) now holds theme/textsize/export/import instead; the header shrunk to just 4 entries. This view was deliberately built as its own screen rather than just trimming icons, because it's the intended future home for login once that ships (see `00_Projektsteuerung/PROJEKTLOG.md` — login itself stays deferred until "the first friend actually joins," per an earlier decision). **Konto/Login — Phase A only (since v1.19.0)**: `renderProfileView()`'s "Konto" section (`accountSectionHTML()`) wires up Firebase Authentication — deliberately **only** Auth, not Firestore data sync. `state`/`persist()`/`loadState()`/`STORAGE_KEY` are completely untouched by this feature; it's purely additive so the existing local-only architecture carries zero risk. Login is optional — the app works exactly as before without ever logging in. The Firebase JS SDK (v12.15.0) is loaded via a separate `<script type="module">` block at the end of the file (CDN ESM imports, e.g. `https://www.gstatic.com/firebasejs/12.15.0/firebase-auth.js`) — the large main `<script>` block deliberately stays a classic script (no app-wide ES-module conversion) and only calls into a thin wrapper exposed on `window.FretzeCloud` (`signInGoogle`, `signInEmail`, `signUpEmail(email, password, displayName)`, `resetPassword`, `signOutUser`, `onAuthChange`) — same isolation principle as `PROXY_URL` for the Gemini worker. The Firebase web config object (`apiKey`, `authDomain`, `projectId`, …) is **not a secret** like the Gemini key — Firebase's own security model relies on Auth + Firestore Security Rules, not on hiding this config, so it's inlined directly in the module script. `currentUser` (ephemeral, never persisted, same convention as `restTimer`/`swapOpenKey`) is set exclusively by the `onAuthChange` callback wired near the bottom of the main script, triggered by a `fretzecloud-ready` custom event the module script dispatches after successful init — if Firebase fails to load (offline on first visit, CDN unreachable), that event never fires, `window.FretzeCloud` stays `undefined`, and every click handler that calls into it checks for that first and shows a toast instead of throwing (same progressive-enhancement pattern as `loadExerciseLibrary()`'s catch-and-swallow). `signUpEmail` resolves with the final `{uid, email, displayName}` after applying an optional chosen display name via `updateProfile()` — the caller uses this resolved value directly (rather than solely relying on `onAuthChange`, whose first fire could race ahead of the `updateProfile()` call and show a stale/empty name). Auth errors are translated to German via `friendlyAuthError(err)` (keyed on Firebase's `err.code`, e.g. `auth/wrong-password`, `auth/email-already-in-use`) — unmapped codes fall back to a generic message including the raw code. **A real bug was found via manual testing, not the Node smoke-test harness** (whose stubbed `document.getElementById` fabricates an element for any id regardless of whether the rendered HTML actually contains it, so it structurally cannot catch a "this element doesn't exist in this view" bug): the `#toast` div used to live only inside the card-view and focus-mode templates, so `showToast()` silently no-op'd everywhere else (Profil, Home, Tagebuch, Bibliothek) — including every login error/success message. Fixed by moving `#toast` into the static HTML shell outside `#app` (alongside `#rest-timer-bar`) and switching its CSS from inline-flow (`margin-top`) to `position: fixed`, so it's now visible from any view. Phase B (Firestore sync) has since shipped in v1.22.0 — see its own paragraph below. `speedfreed079.github.io` was added to Firebase's authorized domains list. Live-verified end-to-end (email/password and Google, including on the real GitHub Pages domain).

**Anzeigename ändern (since v1.20.0)**: a logged-in user can change their display name after the fact (previously only settable at signup) via a new "✎ Ändern" row in `accountSectionHTML()`'s logged-in branch, backed by `window.FretzeCloud.updateDisplayName(name)` (wraps `updateProfile(auth.currentUser, {displayName})`). Same reasoning as `signUpEmail`: the click handler sets `currentUser` directly from the resolved value rather than waiting on `onAuthChange`, since a pure profile-field update doesn't re-fire that listener. Email/password signup now **requires** a display name (client-side check before calling `signUpEmail`) — Google sign-in is unaffected since it always supplies one from the Google account already. `headerSubtitleText()` (the "single obvious edit point for personalizing it to a user's name later" flagged back in the v1.13.0 Profil-view note) now does exactly that: shows `"{possessiveName(displayName)} Trainingsbuch"` when `currentUser.displayName` is set, plain `"Trainingsbuch"` otherwise. `possessiveName(name)` forms the German genitive (plain trailing "s", no apostrophe) but skips adding anything to names already ending in s/x/z (e.g. "Tobias" stays "Tobias" rather than the incorrect "Tobiass") — a small, deliberately simple heuristic, not a full grammar engine.

**Admin-Panel gate (since v1.20.0, functionality still deliberately NOT built)**: a real user-management panel (list/disable/delete accounts) needs the Firebase **Admin** SDK, which must never run in client JS — exposing it there would let any user call those privileged operations themselves, same reasoning as why `GEMINI_API_KEY` lives server-side in the Cloudflare Worker rather than the client. It would need a genuinely privileged backend (e.g. the existing worker extended with admin credentials, or a separate Cloud Function). Phase B (below) has since shipped, so there *is* now real training data in Firestore to potentially manage — but building the actual admin functionality is still a separate, later decision (needs a privileged-backend design, not just a UI). So this version ships only the **gate**: `ADMIN_EMAIL` (currently hardcoded to a single address, no roles system — deliberately not built for a 2-3 person friend group) controls whether an "Admin-Panel" entry point shows in the logged-in Konto section; it leads to `renderAdminView()`, a placeholder screen explaining the above, not yet a real feature.

**Konto/Login — Phase B: Firestore-Datensync (since v1.22.0)**: extends Phase A with real cross-device data sync for logged-in users, while keeping local-only usage (never logged in) completely unaffected. One Firestore document per user at `users/{uid}`, holding the entire `state` object verbatim (same "state is one JSON blob" convention as `persist()`/`loadState()`/`importData()` — no schema redesign, no subcollections) plus a 21st field, `updatedAt` (ms timestamp, stamped synchronously at the top of every `persist()` call). `window.FretzeCloud` (the isolated Firebase module script) gained `saveState(uid, stateObj)` (`setDoc`, JSON-round-tripped first to strip `undefined` values Firestore would otherwise reject) and `loadState(uid)` (`getDoc`, returns `null` if no doc exists yet). `persist()` fires an additional fire-and-forget `saveState()` call (`.catch(() => {})`, never blocks/breaks the local save) after its localStorage write, only `if (currentUser)` — `localStorage` stays the sole always-on store, Firestore is a best-effort mirror on top. `reconcileCloudState(uid)` runs on every `onAuthChange(user)` fire (covers both a fresh login and every page reload with an already-persisted Firebase session, since that callback fires in both cases) — no cloud doc yet means "first login on this account," so the local state is simply pushed up (this doubles as migrating existing local data, no separate migration path needed); if a cloud doc exists, `state.journal` from both sides is **merged** (`mergeJournals()` — dedupe by `timestamp`, chronological sort, union of both arrays) so training entries are never silently lost, while every *other* field follows simple last-write-wins by `updatedAt` (deliberately not a full field-by-field merge — the user was asked and explicitly chose "merge the journal, LWW for the rest" as the pragmatic middle ground over a naive whole-document LWW, which could otherwise silently drop an entire session's data from whichever device pushes second, or a full per-field merge, which would be considerably more code for comparatively low benefit outside the journal). No realtime `onSnapshot` listeners — sync only happens at login/reload time and after each local change while logged in, deliberately, to avoid concurrent-write complexity disproportionate to a 2–3 person friend group. Firestore Security Rules (`users/{uid}` readable/writable only by `request.auth.uid == uid`) live at `03_Tools/firestore/firestore.rules`, **not auto-deployed by any tooling in this repo** (no `firebase-tools`/npm introduced, consistent with the zero-build-tooling philosophy) — must be pasted manually into the Firebase Console whenever changed; see `03_Tools/firestore/README.md`.

**Set-row grid alignment**: `.set-header` and `.set-row` are two *separate* CSS Grid containers, not one shared grid — `auto`-sized columns are computed independently per container, so if their content differs in width (e.g. a header cell holding just an emoji vs. a row cell holding a full button), the columns can visibly drift apart between the header and the actual rows. Fixed pixel widths (not `auto`) are required for any column both grids need to line up on — this is why the 🔥/↑/done columns use fixed px values (`40px 40px 40px` since v1.26.0, up from `32px 32px 24px`) rather than `auto auto auto`; both grids' `grid-template-columns` must be edited together or they'll drift.

**Tap-target sizing (since v1.26.0)**: raised after independent UI/UX feedback from both a Claude role-play session and Gemini converged on the same finding — several controls were well under the ~44px minimum touch-target recommendation, risky during training (often one-handed, sweaty/shaky fingers). `.warmup-toggle`/`.couldmore-toggle`/`.done-btn` grew from ~24–26px to a fixed 40×40px (matching the grid column widths above, `display: flex` + centered content added so the glyph stays centered in the bigger box); `.swap-btn` (ℹ/⟲) and `.home-plan-edit`/`.home-plan-delete` (✏/🗑) grew to `min-width`/`min-height: 44px`. The `.set-row`/`.set-header` grid `gap` was trimmed from 8px to 6px to partially offset the extra fixed-column width eaten from the `1fr` weight/reps inputs on narrow screens — verified via a one-off Playwright screenshot/measurement pass at a 375px viewport (iPhone SE) that nothing overflows horizontally. Several related UI/UX critique items (from the same two feedback rounds) remain open and unprioritized — see `00_Projektsteuerung/MEMORY.md`.

**Rest timer**: `restTimer` (endsAt/duration/intervalId) is module-level, in-memory only — deliberately not part of `state`/`localStorage`. Starts via `startRestTimer()` when a set is marked done (in the `set-done` click handler). Its bar (`#rest-timer-bar`) lives outside `#app` in the static HTML and is updated directly via `renderRestTimer()`, not through the main `render()` pipeline, so it survives full re-renders without extra bookkeeping.

**Modal component (since v1.27.0)**: replaces the native `prompt()`/`confirm()`/`alert()` dialogs (gym add/rename/delete, custom-plan delete, import-overwrite confirmation, import error) with an in-app-styled equivalent. `modal` is an ephemeral module-level variable (never in `state`, same convention as `restTimer`) holding `{ kind: "prompt" | "confirm", title, message, defaultValue, confirmLabel, danger, onConfirm }` — `onConfirm` is a plain function reference, fine since it's never serialized. `openPromptModal(opts)`/`openConfirmModal(opts)` set it and call `renderModal()`; `closeModal()` clears it. Renders into its own `#modal-overlay` element in the static HTML shell, outside `#app` (same reasoning as `#toast`/`#rest-timer-bar` — must be visible from any view, not just card view; see the v1.19.0 `#toast` bug note above for why this matters). `.modal-overlay` is `display: none` by default and only becomes a `display: flex` fullscreen backdrop when non-empty (`:not(:empty)`), so `renderModal()` just needs to set/clear `innerHTML`, no class toggling. Clicking the darkened backdrop or "Abbrechen" closes without side effects; the "Speichern"/positive-confirm button is styled with `--olive`, "Löschen"/destructive-confirm with `--rust`, "Abbrechen" stays neutral — this at least makes the `--rust` dual-use (brand-primary vs. destructive) unambiguous within any single modal, though the broader app-wide reuse of `--rust` for both purposes elsewhere (`home-plan-delete`, `swap-clear-btn`, `rest-timer-skip`) was deliberately **not** globally re-themed, out of scope for this pass. Deliberately no Enter/Escape keyboard handling — would need a fourth global delegated listener type (alongside `click`/`input`/`change`), not proportionate for two buttons.

**KI-gestützter Übungstausch**: `03_Tools/worker/` contains a standalone Cloudflare Worker (deployed at https://eisernes-log-proxy.speedfreed.workers.dev via `wrangler deploy` from that directory, not part of the GitHub Pages deploy) that proxies to the Gemini API — it holds `GEMINI_API_KEY` server-side so the key never reaches the client. The client's `PROXY_URL` constant (top of the script) points at this deployed worker URL; empty string disables the feature gracefully. `fetchAiSuggestions()` calls the worker (25s client-side timeout — real Gemini latency for this model runs ~10s) and stores results in the in-memory (non-persisted) `aiSuggestions` map, keyed like `state.swaps`; results render via `aiSuggestHTML()` inside the existing swap panel and, on selection, flow through the same `swap-select` mechanism as the static `alts`. The worker's CORS `ALLOWED_ORIGIN` is locked to `https://speedfreed079.github.io` — testing locally against it requires temporarily pointing that constant at your local origin and redeploying, then reverting before any real deploy.

**Gemini model choice**: `GEMINI_MODEL` in `03_Tools/worker/src/index.js` is `gemini-3.1-flash-lite` (as of 2026-07-09). Three prior model IDs have gone stale in this project's lifetime, each the same underlying cause (a retired model id) but with different error shapes: `gemini-2.0-flash` was deprecated/shut down 2026-06-01 (`429`/`limit: 0` quota error), then `gemini-2.5-flash-lite` (used 2026-06-01 to 2026-07-09) and — on the very next deploy attempt, same day — its replacement `gemini-2.5-flash` *also* turned out to be retired, both returning a `404` with `"...is no longer available"`. **Lesson learned**: when picking a replacement model, don't trust a docs-page fetch at face value — https://ai.google.dev/gemini-api/docs/models returned a summary claiming both `gemini-2.5-flash`/`gemini-2.5-flash-lite` were still "Stable" when they were provably dead per the live API's own error response. The live error is ground truth; a fetched/summarized docs page is not. After the second failed attempt, `gemini-3.1-flash-lite` was deployed and confirmed working with an actual end-to-end request (real suggestions returned, no error) before being treated as done.

**Google-Drive-Backup (since v1.28.0)**: a third, independent backup path alongside Export/Import and Firestore sync — deliberately **not** connected to the Firebase Auth login at all (no reference to `currentUser`/`window.FretzeCloud`), so it works whether or not the user ever uses "Konto". Uses Google Identity Services (GIS, `https://accounts.google.com/gsi/client`), a modern client-side-only OAuth flow, with its own dedicated OAuth Client ID (`DRIVE_CLIENT_ID`, a new "Fretze Drive Backup" Web-application client registered in the same `fretze-pumpt` GCP project) — deliberately **not** reusing Firebase Auth's own auto-created client, same isolation principle as every other external integration in this app (Gemini worker, Firebase wrapper each get their own independent setup). The Client ID is **not a secret** (unlike the Gemini API key) — security comes from the registered "Authorized JavaScript origins" in Google Cloud Console, not from hiding the ID, same reasoning as the already-public Firebase config. Scope is the narrow `https://www.googleapis.com/auth/drive.file` (the app can only see/manage files it created itself, never the user's whole Drive) — confirmed non-sensitive, so no Google app-verification review is needed regardless of OAuth consent screen publish status. `loadGisScript()` injects the GIS `<script>` tag lazily, only on the first click of "☁ In Drive sichern" (Profil view, Daten section) — a deliberate deviation from Firebase's eager-at-boot loading, since this is an opt-in feature most sessions never touch. `driveAccessToken`/`driveTokenClient`/`driveGisLoadPromise` are ephemeral module-level state (never in `state`/localStorage, same convention as `restTimer`/`modal`) — the access token is short-lived (~1h) and simply re-requested via `getDriveAccessToken()` when absent/expired, no refresh-token handling needed for this implicit-flow model. `backupToDrive()` reuses `buildBackupPayload()` (extracted out of `exportData()` so the JSON shape can't drift between the two backup paths) and uploads it via a plain `fetch()` multipart POST to the Drive v3 REST API (`uploadType=multipart`) — no `gapi` client library, consistent with the zero-build-tooling philosophy. **v1 always creates a new, timestamped file on every click** rather than finding/overwriting the previous backup — simpler, no persisted Drive file-ID needed, and Drive's own version history/trash is enough of a safety net for a few small JSON files from 2-3 users; overwrite-in-place is technically feasible under `drive.file` (its `files.list` is auto-scoped to app-created files, not blocked entirely) but deliberately deferred. Fails gracefully (toast, not a crash) if the GIS script can't load (offline/CDN blocked) or the user cancels/denies the consent popup.

**Secrets hygiene**: `wrangler secret put GEMINI_API_KEY` takes the secret **name** as the CLI argument and prompts separately for the **value** — never pass the key itself as the argument (secret names are plaintext-visible via `wrangler secret list`, unlike values). Always run secret-setting commands in the user's own terminal, never through an agent-relayed shell, so the key value never appears in any transcript/log.
