## Versioning (process rule, active since v1.1.0)

Every change gets a version bump and a `00_Projektsteuerung/PROJEKTLOG.md` entry ‚Äî no exceptions, this was an explicit user request.

- Bump `APP_VERSION` in `index.html` (semantic-ish: PATCH for small fixes, MINOR for features, MAJOR for breaking/rewrite-level changes).
- Bump `CACHE_NAME` in `sw.js` to match (e.g. `fretze-pumpt-v1.2.0`) ‚Äî this is not just bookkeeping, it's what makes the PWA actually update on installed devices (see the auto-update note below). The two must move together.
- Add a dated, versioned entry to `00_Projektsteuerung/PROJEKTLOG.md` describing what changed, before/alongside deploying.

## User-facing documentation (`howto.html`, process rule, active since v1.5.1)

`00_Projektsteuerung/PROJEKTLOG.md` is a developer-facing changelog (for future Claude sessions, not end users). `howto.html` is the separate, plain-German **user-facing** how-to page, linked from the app header ("‚ùì Anleitung") ‚Äî explicit user request: **every change relevant to what the user sees or does must also update `howto.html`**, not just `PROJEKTLOG.md`. Examples of "relevant": a new/changed feature, a UI element that moves or behaves differently, a new settings/toggle. Not relevant: internal refactors, data-model changes with no visible effect, doc/versioning-only commits. `howto.html` reads the same `localStorage` theme key as `index.html` to match dark/sepia automatically, and is listed in `sw.js`'s `APP_SHELL` for offline use ‚Äî keep both in sync if it's ever renamed. **Sections are collapsible `<details>`/`<summary>` since v1.56.0** (user feedback: 24 always-expanded `.card` sections in a row felt overwhelming) ‚Äî every section except the un-headed intro blurb is now `<details class="card" id="...">` with the `<h2>` wrapped in `<summary>`, all closed by default, no JS needed for the collapse/expand itself (native HTML). The one thing that DID need a few lines of JS (in the existing inline `<script>` at the bottom, next to the theme-sync logic): the top-of-page TOC links to `#anchor` ids that now live on closed `<details>` elements ‚Äî a native browser anchor-jump scrolls to a closed section without opening it, so `openTargetDetails()` runs on load and on `hashchange` to explicitly set `.open = true` and re-`scrollIntoView` the target. When adding a new section, follow the existing `<details class="card" id="foo"><summary><h2>...</h2></summary>...</details>` pattern ‚Äî a plain `<div class="card">` (no `<details>`) only stays correct for content with no heading/TOC entry, like the current intro blurb.

## Commands

There is no build, lint, or test tooling ‚Äî no `package.json`, no dependencies. The app is plain HTML/CSS/JS.

- **Develop**: edit `index.html` directly and reload in a browser.
- **Test PWA/offline behavior**: the service worker (`sw.js`) only registers on a real HTTP origin, not `file://`. Serve the folder locally, e.g. `npx serve` or `python -m http.server`, then open it.
- **Deploy**: pushing to `main` on GitHub is the deploy ‚Äî GitHub Pages serves this repo directly (no `gh-pages` branch, no build step). Live at the repo's Pages URL. Only push to `main` when explicitly asked to deploy.
- **Deploy the Gemini proxy worker**: `cd 03_Tools/worker && wrangler deploy` (requires `wrangler secret put GEMINI_API_KEY` once beforehand). This is separate from the GitHub Pages deploy ‚Äî the worker lives on Cloudflare, not in this repo's Pages site.

## Architecture

Everything lives in one file, `index.html`: inline `<style>` block plus a single inline `<script>` at the bottom (~680 lines total). No framework, no bundler. Supporting files are `manifest.json` (PWA metadata), `sw.js` (cache-first service worker caching the app shell), `icon-*.png`, `logo-mark-dark.png`/`logo-mark-sepia.png` (header logo, transparent PNGs; see Ordnerstruktur above for the source file), and `howto.html` (standalone user-facing how-to page, see the User-facing documentation section below). All must stay in the app shell list in `sw.js` (`APP_SHELL`) ‚Äî a new asset referenced from `index.html` but missing there won't be available offline.

**Header logo is theme-aware, unlike the app icons**: `render()` picks `logo-mark-dark.png` or `logo-mark-sepia.png` based on `state.theme`, mirroring the existing `theme-color`-meta swap. Two separate PNGs exist (rather than one raster recolored to the shared `--rust`) because a single color didn't read well in both contexts: dark theme needed a brighter/more saturated orange (`#F97316`) for the fine linework to stay legible against the near-black `--bg`, while sepia keeps the theme's own `--rust` (`#A8461E`), which already contrasts fine on the light background. The app-icon PNGs (`icon-192.png`/`icon-512*.png`) are NOT theme-aware ‚Äî OS home screens don't know about in-app theme state, so there's necessarily one fixed appearance regardless of `state.theme`. Since v1.14.0 that fixed appearance is the **sepia** styling (cream `#F2E8D5` background, `logo-mark-sepia.png`'s rust linework, generated at a much higher fill fraction ‚Äî ~86% of the canvas for `icon-192.png`/`icon-512.png`, ~74% for the `icon-512-maskable.png` safe zone ‚Äî than the original export, which left a lot of empty padding and read as "too small" per user feedback) ‚Äî chosen because sepia is also the app's actual default theme (`loadState()`'s fallback). `manifest.json`'s `background_color`/`theme_color` and `index.html`'s static initial `<meta name="theme-color">` were updated to the same sepia palette (`#F2E8D5`/`#A8461E`) to match, avoiding a dark splash-screen flash against the now-cream icon. Regenerating these icons requires re-deriving the logo's tight bounding box from `logo-mark-sepia.png` (or `-dark.png`) and re-detecting the existing rounded-corner cutout shape from the *previous* icon file as a mask ‚Äî see the (uncommitted, session-scratch) Python regeneration script for the exact approach if this needs doing again.

**PWA update mechanism**: `sw.js` uses a cache-first strategy, which means an installed/home-screen copy will keep serving whatever it last cached until the service worker itself gets updated ‚Äî browsers only re-check `sw.js` for byte-level changes, they don't know `index.html` changed on their own. That's why `CACHE_NAME` must change every release (see Versioning below): a different string makes `sw.js`'s bytes differ, which is what triggers the browser to install the new worker (`skipWaiting()` + `clients.claim()` in the code make it take over immediately), delete the old-named cache, and re-fetch the app shell fresh. Skipping the `CACHE_NAME` bump on a release means installed devices silently keep the old version.

**Naming stability**: `STORAGE_KEY` (`eisernes-log-v1`) and the Cloudflare Worker name (`eisernes-log-proxy`) were deliberately *not* renamed during the 2026-07-08 rebrand to "Fretze pumpt", nor during the 2026-07-09 shortening to "Fretze" ‚Äî changing `STORAGE_KEY` would wipe every user's existing `localStorage` data, and renaming the deployed worker has no user-facing benefit. Don't rename these without explicitly flagging the data-loss risk to the user first.

**Data model** (top of the script):
- **Multiple plans (since v1.5.0, 5th added in v1.21.0, 6th in v1.24.0)**: `PLANS` is an array of `{ id, label, days, exercises }` ‚Äî currently `ppl` (`PPL_DAYS`/`PPL_EXERCISES`, the original 6-day split), `upperLower` (`UL_DAYS`/`UL_EXERCISES`), `fullbody` (`FB_DAYS`/`FB_EXERCISES`), `broSplit` (`BRO_DAYS`/`BRO_EXERCISES`), `calisthenics` (`CALI_DAYS`/`CALI_EXERCISES`, a 4-day bodyweight-only Upper/Lower split, sourced from `01_Recherchen/01_Trainingsplaene/Zuhause-Calisthenics-Trainingsplan.md`), `baender` (`BAND_DAYS`/`BAND_EXERCISES`, a 4-day Upper/Lower split for the "SmartWorkout Elite" resistance-band set ‚Äî see the dedicated paragraph below) ‚Äî all sourced from `01_Recherchen/01_Trainingsplaene/`. `DAYS`/`EXERCISES` are `let` (not `const`): they're mutable pointers to the *active* plan's data, reassigned by `applyPlan(planId)`. Every other function in the file (`render()`, event handlers, `getRecommendedDay()`, ‚Ä¶) reads `DAYS`/`EXERCISES` normally and doesn't need to know plans exist. **`PLANS` itself stays fixed to the 6 built-ins** (since v1.11.0) ‚Äî user-created plans live in `state.customPlans` instead, and `getAllPlans()` (`[...PLANS, ...state.customPlans]`) is what `applyPlan()`, the plan dropdown, and the home screen actually iterate; don't read `PLANS` directly anywhere that should also see custom plans.
- **Day/exercise IDs are globally unique across all plans** (e.g. PPL uses `pushA`/`pa1`, Upper/Lower uses `ulUpperA`/`uua1`, never overlapping) ‚Äî this is *the* mechanism that makes "switch plans without losing data" work: `state.sessions`/`history`/`swaps`/`notes`/`lastWeights` stay flat objects keyed by these IDs, so switching `state.currentPlan` never overwrites another plan's entries; nothing needed migrating. Don't reuse an ID across plans.
- `DAYS` entries have an optional `type` (`"push"`/`"pull"`/`"legs"`) used to look up `WARMUP`/`STRETCHING` content per day-type. Only assigned where the research content genuinely applies (e.g. Upper/Lower's "Lower" days get `"legs"`, Bro-Split's "Schultern"/"Arme" days get none) ‚Äî `warmupSectionHTML()`/`stretchSectionHTML()` return `""` when there's no matching entry, so a missing `type` just means no section shows, not a crash.
- `EXERCISES`: keyed by day id, each an array of exercises with `id`, `name`, `sets`, `reps` (target range), optional `alts` (swappable alternative exercise names), `core` (boolean flag for core exercises), `compound` (boolean, since v1.12.0 ‚Äî flags the canonical "big lifts": squat, deadlift/RDL, flat bench, overhead press, pull-up, barbell row, regardless of exact equipment; drives the higher progression percentage in `computeSuggestedWeight`), and `libraryId` (optional, since v1.16.0 ‚Äî see below). This is the content to edit when changing a training program itself. Only the `ppl` plan's exercises have populated `alts`; the 4 newer built-in plans (`upperLower`/`fullbody`/`broSplit`/`baender`) default to `alts: []` (no researched alternatives yet) ‚Äî `calisthenics` is the one exception, see its own paragraph below. Custom (user-built) plan exercises get no `alts`/`core`/`compound` ‚Äî always the defaults/standard progression rate ‚Äî but *can* have a `libraryId` if built from a database-picked exercise name.
- **√úbungsdatenbank / `exerciseLibrary` (since v1.16.0)**: exercise *knowledge* (muscles/steps/mistakes/safety, optionally an `image`) lives in `exercise-library.json` (repo root, sibling to `manifest.json`) ‚Äî deliberately **not** inlined in `index.html`, so the database can grow without bloating the app bundle or needing an `APP_VERSION` bump per new exercise. Keyed by a stable **content ID** (a slug derived from the exercise name, e.g. `flachbankdruecken-kurzhantel`) that's independent of any plan ‚Äî this is the key architectural difference from the old `EXERCISE_INFO` (which was keyed by plan-slot `ex.id`, so the same real exercise appearing in multiple plans would've needed its explanation duplicated per plan). Exercise objects link to it via the optional `libraryId` field; `exerciseInfoPanelHTML(ex)` (the single function building the ‚Ñπ-panel markup, called from card view/focus mode/library view) resolves it as `exerciseLibrary[ex.libraryId]` ‚Äî no `libraryId`, or a `libraryId` the library doesn't (yet) have, both degrade gracefully to no ‚Ñπ button, same as the old missing-`EXERCISE_INFO`-entry behavior. Loaded once at boot via `loadExerciseLibrary()` (`async`, called *after* the first synchronous `render()` so the app doesn't wait on the fetch to become usable ‚Äî `exerciseLibrary` starts as `{}` and a failed/offline fetch just leaves it that way, caught and swallowed, no crash). `sw.js` needed **no changes** for this: its existing `fetch` handler already caches any same-origin GET response after first fetch, not just `APP_SHELL` entries, so `exercise-library.json` (and future per-exercise images) get cached opportunistically without bloating the explicit offline app-shell list. All 4 built-in plans now have `libraryId`s populated (`ppl` migrated 1:1 from the old `EXERCISE_INFO`, with one real duplicate ‚Äî "Beinstrecker" existed twice, verbatim, under two plan-slot IDs ‚Äî collapsed into a single library entry both slots now point to; Upper/Lower/Ganzk√∂rper/Bro-Split added in v1.17.0 from a second Deep Research batch of 37 exercises, `01_Recherchen/02_Uebungserklaerungen/Uebungsdatenbank-UL-FB-BroSplit.md`, parsed and merged the same way). The library-first design pays off here: the same real exercise repeated across multiple plans (e.g. "Kniebeugen mit der Langhantel" in Upper/Lower, Ganzk√∂rper, and Bro-Split) shares **one** library entry rather than being researched/stored three times. Custom plan exercises still get no `libraryId` unless built from a database-picked name (see the plan-builder note above) ‚Äî same graceful gap as before. The plan-builder's exercise-name `<datalist>` (`planBuilderExerciseNamesDatalist()`) is sourced from `exerciseLibrary` (not raw plan-exercise names) specifically so picking a suggested name always yields a working ‚Ñπ explanation ‚Äî `libraryIdForName(name)` does the reverse lookup and gets auto-attached to a draft exercise when its name exactly matches a library entry; free-text names simply get no `libraryId`, exactly like today's gap.
- `WARMUP`/`STRETCHING`/`STRETCHING_NOTE`: keyed by day `type`. Sourced from `01_Recherchen/03_Aufwaermen/` and `01_Recherchen/04_Stretching/`; rendered as collapsible sections above/below the exercise cards. None of this ‚Äî nor the exercise-info panel ‚Äî is part of `state`/`localStorage`; open/closed state (`infoOpenKey`/`warmupOpen`/`stretchOpen`) is ephemeral like `swapOpenKey`. **`WARMUP` revised in v1.40.0** (user feedback: the original v1.4.0 content felt too involved) following a fresh Deep Research pass (`01_Recherchen/03_Aufwaermen/Wissenschaftliches Aufw√§rmen im Krafttraining.md` ‚Äî the user accidentally ran the research prompt twice and got two diverging syntheses; this one was picked after the user chose explicitly between the conflicting recommendations via `AskUserQuestion`). Changes: `general`'s cardio-equipment recommendation is now fixed per day-type (Crosstrainer/push, Ruderergometer/pull, Fahrradergometer/legs ‚Äî movement-specificity argument) rather than the old rowing-only default; `activation` (resistance-band exercises) is now **conditionally omitted for `legs`** (`activation: []` ‚Äî the research found glute pre-activation redundant given a heavy squat's own recruitment demand, but kept it for push/pull's rotator-cuff/scapular stabilization role) ‚Äî `warmupSectionHTML()` was updated to skip rendering the "Aktivierung" sub-section entirely when the array is empty, rather than showing an empty heading; `rampSets` simplified from 4 stages of the day's *1RM* to 3 stages of the day's *planned working weight* for the first compound lift only (30-40%/60%/80%, matching the app's existing "arbeitsgewicht"-relative wording in `warmupSectionHTML()` ‚Äî no template change needed there).
- **`calisthenics` plan's `alts` repurposed for progression, not lateral swaps (since v1.21.0)**: every other plan's `alts` lists genuinely different alternative exercises (same muscle group, different movement) for the ‚ü≤ swap feature. The calisthenics research instead frames progression as **difficulty variants of the same exercise** (e.g. knee push-ups ‚Üí push-ups ‚Üí archer push-ups), since there's no external weight to add for pure bodyweight training. Rather than building a dedicated progression UI, `CALI_EXERCISES` populates `alts: [easierVariant, harderVariant]` and reuses the existing swap panel as-is ‚Äî a deliberate, pragmatic repurposing of an existing mechanism rather than new architecture. **All 16 of the plan's exercises have a `libraryId` since v1.39.0**: 2 reuse the Klimmz√ºge (pull-up/chin-up) entries already shared with Upper/Lower/Ganzk√∂rper/Bro-Split, "H√§ngendes Beinheben" reuses its existing shared entry, and the remaining 13 (the push-up family, Pike Push-ups, Pistol Squats, Nordic Hamstring Curls, Sliding Leg Curls, Bulgarian Split Squats, Inverted Rows, RKC Plank) came from a dedicated follow-up research batch (`01_Recherchen/02_Uebungserklaerungen/Biomechanischer √úbungskatalog Calisthenics.md`), parsed and merged the same way as UL/FB/BroSplit in v1.17.0 ‚Äî `calisthenics` was the last plan with a library gap, now closed.

**State & persistence**: a single in-memory `state` object (`currentDay`, `currentPlan`, `sessions`, `history`, `swaps`, `notes`, `lastWeights`, `lastCompletedDay`, `lastCompletedDate`, `theme`, `textSize`, `gyms`, `currentGymId`, `gymLastWeights`, `gymHistory`, `journal`, `customPlans`, `planDraft`, `weightSuggestions`, `gymWeightSuggestions`) is loaded from and persisted to `localStorage` under key `eisernes-log-v1` (debounced ~300ms via `persist()`). `textSize` (`"normal"`/`"gross"`/`"xgross"`) drives the `--text-zoom` CSS variable (via `data-textsize` on `<html>`, same pattern as `data-theme`) applied as `zoom` on `#app`/`#rest-timer-bar` ‚Äî scales the whole UI, not just font-size declarations, so no per-rule CSS changes were needed. `zoom` needs Firefox 126+ (mid-2024); older Firefox just silently ignores it (graceful, no crash). `applyPlan(state.currentPlan)` must run after `state` is loaded/replaced (on initial load and after Import) so `DAYS`/`EXERCISES` point at the right plan before anything renders ‚Äî both call sites also snap `state.currentDay` back to `DAYS[0].id` if it doesn't belong to the now-active plan. There is no backend ‚Äî the Export/Import buttons in the header serialize/restore this state as a JSON file, and that's the only backup mechanism.

**Rendering**: no virtual DOM ‚Äî `render()` rebuilds `#app`'s `innerHTML` from `state` on every state change (day switch, set marked done, warmup-set toggle, swap, finish session), always through the shared `setAppHTML(html)` helper (since v1.30.0) rather than assigning `innerHTML` directly ‚Äî every one of `render()`'s 7 early-return view branches plus the card-view fallthrough at the end all call it. `setAppHTML()` adds a 150ms CSS fade-in (`.fade-in` class + `app-fade-in` keyframe) on every view swap ‚Äî removes the class, forces a reflow (`void app.offsetWidth`), re-adds it, so a render mid-animation restarts cleanly instead of glitching. Purely cosmetic, no interaction with `state`. One exception to `render()` itself: raw text inputs (set weight/reps, notes) update `state` directly in an `input` listener without calling `render()`/`setAppHTML()`, to preserve focus/cursor position while typing; only `persist()` runs there ‚Äî this is also why the Cloud-Sync-Statusanzeige (see below) deliberately avoids triggering a full `render()` on every status change.

**Events**: everything is delegated through three top-level listeners (`click`, `input`, `change`) on `document`, dispatching on `data-role` attributes (e.g. `data-role="set-done"`, `data-role="swap-toggle"`, `data-role="finish"`) rather than per-element handlers. New interactive elements should follow this pattern: give them a `data-role` (plus `data-day`/`data-exid`/`data-idx` as needed) and add a branch in the relevant delegated listener.

**Session flow**: `finishSession(dayId, wellbeing, sessionNote, dateOverride)` records the last completed set per exercise into `state.history` (shown as "Letztes Mal: ..."), snapshots current weights into `state.lastWeights` (used by `buildInitialSets` to prefill next time), resets the day's sets, and stamps `lastCompletedDay`/`lastCompletedDate` (used by `renderTodayBanner()`/`getRecommendedDay()` to suggest the next day in the PPL rotation on the home banner). **Since v1.33.0** the `finish` click handler (shared by card view and focus mode, `data-role="finish"`) no longer calls `finishSession()` directly ‚Äî it first opens a `kind: "choice"` modal (üòä/üòê/üòñ + "√úberspringen", **plus an optional note textarea since v1.34.0**) via `openChoiceModal()`, and `finishSession(dayId, wellbeing, sessionNote, dateOverride)` is only invoked from that modal's `onConfirm` callback. Both the rating and the note are entirely optional and non-blocking ‚Äî skipping still finishes the session, just without those values on the journal entry (`sessionNote` is trimmed, empty string becomes `null`). **`dateOverride` since v1.50.0 ("Training nachtragen")**: the same choice modal now always shows an `<input type="date">` (defaulting to today, `max` capped at today) via `openChoiceModal`'s new `includeDate`/`defaultDate` options; `finishSession` builds a `refDate` from it via the shared `dateOverrideToRefDate(dateOverride)` helper, used for both the journal entry's `date`/`timestamp` and `state.lastCompletedDate` ‚Äî this is what actually makes backdating work, since `timestamp` is what `getWeekStart()`-based streak/volume/PR calculations bucket by. **`dateOverrideToRefDate()` combines the chosen calendar date with the real current time-of-day, not a fixed noon** ‚Äî an earlier version stamped a fixed `T12:00:00`, which meant every normal same-day finish (the date field defaults to today and is usually left untouched) produced an *identical* timestamp for multiple sessions logged the same day. That collided with `timestamp` being the de facto unique entry ID (`mergeJournals()`'s cross-device dedup key, and since v1.51.0 also `journal-delete-entry`'s lookup key) ‚Äî found via a Playwright test for the delete feature below, where deleting one of three same-day entries deleted all three. Fixed before it reached most users; still worth knowing if `timestamp` uniqueness ever seems to misbehave elsewhere. No dedicated "add a past workout" screen exists or is needed: any day tab within the active plan can be opened and filled in regardless of whether it's "today's" day (`state.sessions` is independent per day-id and only resets on that day's own finish, so there's no collision risk with a same-day session in progress elsewhere) ‚Äî backdating a forgotten-to-finish session is just "open that day's tab, the entered sets are still sitting there, finish it with the date pulled back." **Set-row grid alignment**: `.set-header` and `.set-row` are two *separate* CSS Grid containers, not one shared grid ‚Äî `auto`-sized columns are computed independently per container, so if their content differs in width (e.g. a header cell holding just an emoji vs. a row cell holding a full button), the columns can visibly drift apart between the header and the actual rows. Fixed pixel widths (not `auto`) are required for any column both grids need to line up on ‚Äî this is why the üî•/‚Üë/done columns use fixed px values (`40px 40px 40px` since v1.26.0, up from `32px 32px 24px`) rather than `auto auto auto`; both grids' `grid-template-columns` must be edited together or they'll drift. **KG/Wdh. column ratio changed in v1.55.0**: was `1fr 1fr` (equal split) for the two `.set-input` columns; on narrow real devices this truncated multi-digit/decimal weights (e.g. "120" or "22,5" scrolling out of view inside the input, per a user screenshot ‚Äî not reproducible at the app's usual desktop-width Playwright testing). Changed to `1.5fr 0.9fr` (weight gets meaningfully more room than reps, which are almost always 1-2 digits) plus the shared 22px index column trimmed to 20px and `.set-input` padding trimmed from `7px 8px` to `7px 5px` ‚Äî `min-width: 0` was also added to `.set-input` since grid items default to `min-width: auto`, which can silently override `fr`-based sizing and prevent an input from actually shrinking/growing as the ratio intends.

**Card-header flex overflow (min-width:0 gotcha, hit twice)**: the same `min-width: auto`-on-a-flex-child issue documented under "Layout gotcha" below (focus mode, v1.32.0) turned out to also affect the *card-view* exercise header ‚Äî `.card-top`'s title wrapper was a bare inline `style="flex:1;"` with no `min-width: 0`, so a two-line-wrapped exercise name (e.g. "Hackenschmidt / Beinpresse") could refuse to shrink enough, pushing the ‚Ñπ/‚ü≤/‚úé button row (`flex-shrink: 0`) partially outside the card's right edge. Fixed in v1.55.0 by adding `min-width: 0` to that wrapper (both the card-view and physio-view instances ‚Äî `grep 'style="flex:1;"'` found both) and to `.card-title-row h3` itself (`overflow-wrap: break-word` added there too). **Lesson**: any `flex: 1` sibling next to a `flex-shrink: 0` block needs an explicit `min-width: 0` unless you've verified its content can never need to shrink below its own intrinsic width ‚Äî worth checking for this pattern before adding a third instance somewhere.

**Trainingsmodus title/buttons stacked instead of side-by-side (v1.55.0)**: previously `.card-top`'s title (`flex: 1; min-width: 0`) sat beside the ‚Ñπ/‚ü≤/‚úé button row (`flex-shrink: 0`) on one line, same layout as card view. For a long-but-normal exercise name (e.g. "Chest-Supported Row") at the Trainingsmodus's larger 32px title font, the leftover width next to 3 buttons was sometimes too narrow even for a single word like "Supported", forcing `overflow-wrap: break-word` to hyphenate mid-word ("SUPPO-RTED") ‚Äî ugly, and *not* fixable by `min-width: 0` alone, since `min-width: 0` is exactly what lets the title keep shrinking instead of wrapping the buttons to their own line (confirmed by testing: adding `flex-wrap: wrap` to `.card-top` alone changed nothing, because the title's own flexibility absorbed all the pressure first). Real fix: a `.focusmode-top` modifier class (`flex-direction: column; align-items: stretch`) makes the Trainingsmodus instance of `.card-top` stack the title block above a new `.focusmode-top-actions` button row instead of placing them side by side ‚Äî the title now always gets the full card width, and testing confirms this reads fine for short names too, not just the long-name edge case. Card view's `.card-top` itself is untouched (still row layout) ‚Äî only the Trainingsmodus instance uses the new modifier class.

**Google-Drive-Backup (since v1.28.0)**: a third, independent backup path alongside Export/Import and Firestore sync ‚Äî deliberately **not** connected to the Firebase Auth login at all (no reference to `currentUser`/`window.FretzeCloud`), so it works whether or not the user ever uses "Konto".
