## v1.22.0 — 2026-07-10 (Teil 45): Firestore-Sync (Login Phase B)

- Nach eigener Planungsrunde (Explore + Plan-Agent, vom Nutzer freigegeben) umgesetzt: eingeloggte Nutzer bekommen jetzt echten Datensync statt nur reiner Anmeldung (Phase A, v1.19.0/v1.20.0). Nicht eingeloggte Nutzung bleibt unverändert rein lokal.
- **Datenmodell**: ein Firestore-Dokument pro Nutzer (`users/{uid}`), enthält den kompletten `state`-Blob unverändert plus ein neues 21. Feld `updatedAt` (ms-Timestamp, gestempelt bei jedem `persist()`-Aufruf) — bewusst kein Redesign, keine Subcollections, spiegelt exakt die bestehende "state ist ein JSON-Blob"-Konvention von `persist()`/`loadState()`/`importData()`. `updatedAt` bekommt an allen drei Stellen, die `state` feldweise rekonstruieren, denselben expliziten Default wie die anderen 20 Felder.
- **`window.FretzeCloud`** (das isolierte `<script type="module">` aus Phase A) um `saveState(uid, stateObj)` (Firestore `setDoc`, JSON-Rundreise vorher sanitisiert `undefined`-Felder, die Firestore anders als `JSON.stringify` ablehnt) und `loadState(uid)` (Firestore `getDoc`, `null` wenn kein Dokument existiert) erweitert — neuer `firebase-firestore.js`-Import, neue `getFirestore(app)`-Instanz, sonst unverändert isoliert vom Haupt-Script.
- **`reconcileCloudState(uid)`** (neu, Haupt-Script): läuft bei jedem Login und bei jedem Seiten-Reload mit bereits bestehender Firebase-Session (derselbe `onAuthChange`-Callback deckt beide Fälle ab, eine Zeile im bestehenden Listener ergänzt). Kein Cloud-Dokument vorhanden → lokalen Stand hochladen (deckt "bestehende Daten beim ersten Login migrieren" automatisch mit ab, kein separater Migrationspfad nötig). Cloud-Dokument vorhanden → **Tagebuch (`state.journal`) wird über beide Geräte zusammengeführt** (`mergeJournals()`, Dedupe per `timestamp`, chronologisch sortiert — Trainingseinträge sollen nie stillschweigend verloren gehen), alle anderen Felder folgen einfachem Last-Write-Wins nach `updatedAt`.
- **Nutzer-Rückfrage zur Merge-Strategie**: naives Ganzes-Dokument-LWW könnte bei echt gleichzeitiger Offline-Nutzung auf zwei Geräten im Ausnahmefall ein komplettes Training stillschweigend verwerfen. Der Nutzer hat sich explizit für die Tagebuch-Sonderbehandlung entschieden (statt reinem Ganzes-Dokument-LWW oder einer vollen Feld-für-Feld-Merge-Maschine) — schützt die schmerzhaftesten Daten, ohne die Sync-Logik grundlegend zu verkomplizieren.
- **`persist()`** stempelt jetzt `state.updatedAt = Date.now()` synchron bei jedem Aufruf und pusht (fire-and-forget, `.catch(() => {})`, nur wenn eingeloggt) den State zusätzlich zu Firestore, huckepack auf demselben bestehenden 300ms-Debounce-Timer — `localStorage` bleibt der einzige immer verfügbare Speicher, der Firestore-Push blockiert/bricht bei Offline nie den lokalen Save.
- **Security Rules**: neue Datei `03_Tools/firestore/firestore.rules` (`allow read, write: if request.auth != null && request.auth.uid == uid`) sperrt jedes Nutzerdokument auf seinen eigenen Besitzer. Bewusst kein `firebase-tools`/npm eingeführt (Repo hat null Build-Tooling) — die Regeln müssen manuell in der Firebase Console (Firestore Database → Rules → Publish) eingefügt werden, siehe `03_Tools/firestore/README.md`. **Wichtiger offener Action-Punkt vor dem ersten produktiven Login mit dieser Version**: diese Regeln müssen tatsächlich in der Console veröffentlicht werden, sonst wäre jedes Nutzerdokument für jeden eingeloggten Nutzer lesbar/schreibbar — kein "kann warten"-Punkt.
- Bewusst **keine** Realtime-`onSnapshot`-Listener in dieser Version — Sync passiert nur bei Login/Reload und nach jeder lokalen Änderung während eingeloggt, nicht kontinuierlich im Hintergrund (unverhältnismäßige Komplexität für eine 2-3-Personen-Freundesgruppe).
- `howto.html` (Konto-Abschnitt: automatischer Datenabgleich inkl. ehrlichem Hinweis auf die Tagebuch-Sonderbehandlung und das LWW-Restrisiko bei allem anderen; Backup-Abschnitt: erste Zeile angepasst, da "kein Konto, kein Server" nicht mehr für eingeloggte Nutzer stimmt) aktualisiert.

## v1.2.3 — 2026-07-08 (Teil 9): Header bricht auf schmalen Screens sauber um

- Nutzer meldete: auf dem Handy erschien unter dem Titel nur noch "Push · Pull · ", der Rest der Subzeile war abgeschnitten (v1.2.1-Ellipsis-Fix hat den Titel-Container nicht ausreichend Platz gelassen, weil `flex: 1` mit Flex-Basis 0 der Actions-Leiste (Theme-Toggle + Export + Import) fast die ganze Zeile überlässt, bevor überhaupt "gewrapped" würde).
- Ursache: `.header-row` ist ein Flex-Container mit `justify-content: space-between`; der Titel-Block hatte `flex:1` (Basis 0), wodurch er theoretisch immer "passt" und der Browser nie auf eine zweite Zeile umbricht — er schrumpft stattdessen bis fast auf 0, egal wie eng es wird.
- Fix: neue Media Query `@media (max-width: 480px)` stellt `.header-row` auf `flex-direction: column` um, sodass Titel/Subzeile und die Actions-Leiste auf schmalen Screens sauber übereinander statt nebeneinander stehen. Auf breiteren Screens (Tablet/Desktop) bleibt das bisherige Nebeneinander-Layout.
- Zusätzlich: dem Nutzer erklärt, dass die installierte PWA auf dem Handy einmal komplett geschlossen (aus der App-Übersicht wischen) und neu geöffnet werden muss, damit der neue Service Worker greift — reines "Aktualisieren" innerhalb der offenen App reicht nicht, weil die bereits geladene Seite von der alten `sw.js`-Instanz kontrolliert bleibt, bis eine echte Neuladung/Neustart passiert.
